Context Map Patterns (DDD)

Índice

  1. Nomes e abreviações
  2. Descrição dos Patterns
  3. C4 Model: Diagrama de Container com Context Map Patterns
  4. Cheat Sheet

Os Context Map Patterns descrevem como Bounded Contexts se relacionam entre si em um sistema distribuído. Enquanto o Bounded Context delimita onde um modelo é válido, o Context Map explicita as integrações, dependências e relações organizacionais entre esses contextos.

Esses padrões não tratam apenas de integração técnica (APIs, eventos, bancos), mas principalmente de alinhamento entre times, autonomia e acoplamento. Cada escolha de padrão implica trade-offs claros entre:

  • Acoplamento vs. independência
  • Velocidade de entrega vs. controle do modelo
  • Custo de integração vs. proteção do domínio
  • Governança organizacional vs. flexibilidade local

Na prática, os padrões ajudam a responder perguntas como:

  • Quem controla o modelo compartilhado?
  • Quem se adapta a quem?
  • Onde ocorre tradução entre modelos?
  • Qual o nível aceitável de dependência entre times?

Nomes e abreviações

  • Open-Host Service (Serviço de Host Aberto): OHS
  • Anticorruption Layer (Camada Anticorrupção): ACL
  • Conformist (Conformista): CF
  • Shared Kernel (Kernel Compartilhado): SK
  • Customer-Supplier (Cliente-Fornecedor): CUS (Customer / Cliente) and SUP (Supplier / Fornecedor)
  • Separate Ways (Caminhos Separados): SW
  • Published Language (Linguagem Publicada): PL
  • Big Ball Of Mud (Grande Bola de Lama): BBoM
  • Partnership (Parceria): not abbreviate

Descrição dos Patterns

  • Open-Host Service (Serviço de Host Aberto) (OHS): Uma API pública fornecida por um time upstream que atende múltiplos consumidores. Evita APIs ponto-a-ponto ao oferecer um modelo comum e de propósito geral.
  • Anticorruption Layer (Camada Anticorrupção) (ACL): Implementado por um time downstream, este padrão traduz e transforma um modelo externo de outro time ou sistema para o modelo interno do time. Protege o modelo de domínio central e restringe o acoplamento a uma única camada de adaptação.
  • Conformist (Conformista) (CF): Um time downstream adere integralmente ao modelo upstream sem qualquer transformação. Embora tenha maior acoplamento que um ACL, é útil pela simplicidade, cumprimento de contratos ou quando o contexto upstream lida com cálculos altamente regulados (como avaliação de garantias imobiliárias) ou agregações especializadas.
  • Shared Kernel (Kernel Compartilhado) (SK): Dois times compartilham fisicamente um subconjunto do modelo de domínio, como JARs compartilhados ou um banco de dados compartilhado. Isso introduz alto grau de acoplamento e exige coordenação significativa. É recomendado quando um time gerencia múltiplos contextos com linguagem sobreposta, mas deve ser evitado se os times estiverem em situação competitiva.
  • Partnership (Parceria): Uma relação cooperativa, puramente organizacional, onde os times estabelecem planejamento coordenado e gestão conjunta da integração. Este padrão é altamente recomendado para times que dependem de um Shared Kernel.
  • Customer-Supplier (Cliente-Fornecedor): Um padrão organizacional onde o time downstream atua como “cliente” e ganha influência sobre o planejamento, tarefas e prioridades do time upstream (“fornecedor”).
  • Separate Ways (Caminhos Separados) (SW): Um bounded context opera sem conexões com outros. É escolhido quando a integração é muito cara ou demorada, sendo uma abordagem interessante para MVPs ou cenários de alta incerteza de mercado.
  • Published Language (Linguagem Publicada) (PL): Uma linguagem bem documentada compartilhada entre contextos, onde cada bounded context pode traduzir de e para ela. Frequentemente definida por um consórcio e combinada com um Open-Host Service; exemplos incluem iCalendar ou vCard.
  • Big Ball Of Mud (Grande Bola de Lama) (BBoM): Um sistema (ou parte dele) que se tornou caótico devido a modelos misturados e limites inconsistentes. Times downstream devem sempre usar um Anticorruption Layer ao integrar com um Big Ball of Mud para evitar que o modelo ruim contamine seu próprio contexto.

C4 Model: Diagrama de Container com Context Map Patterns

Uma forma de comunicar o padrão de relacionamento é adicionar um identificador em uma das pontas da conexão dos diagramas C4.

Nos diagramas Mermaid podemos inserir o tipo de relacionamento (CF e OHS) na legenda do Rel, exemplo:

1

Rel_D(checkout, email, "[CF] Sends confirmations [OHS]", "async")

C4Container
title E-Commerce System - Diagrama de Container + Context Map Patterns

%% Legend  
%% ACL = Anti-Corruption Layer  
%% OHS = Open Host Service  
%% CF = Conformist
%% PL = Published Language



System_Boundary(ecommerce, "E-Commerce System") {
    Container(checkout, "Checkout Service", "Web App", "Cart, checkout flow, payments")
    Container(inventory, "Inventory Service", "Web App", "Stock levels, reservation")
    
    ContainerDb(checkout_db, "Checkout DB", "Database", "Orders, carts")
    ContainerDb(inventory_db, "Inventory DB", "Database", "Stock, warehouse")
}


System_Ext(stripe, "Stripe", "Payment API")
System_Ext(email, "Email Service", "Notifications")


Person(customer, "Customer", "Shopper")
Rel(customer, checkout, "Uses", "https")

Rel_R(checkout, inventory, "OrderPlaced event [PL]", "async")
Rel(checkout, checkout_db, "Reads/Writes")
Rel(inventory, inventory_db, "Reads/Writes")



BiRel(checkout, stripe, "[CF] Processes payments [OHS]", "sync")
Rel_D(checkout, email, "[CF] Sends confirmations [OHS]", "async")



UpdateLayoutConfig($c4ShapeInRow="3", $c4BoundaryInRow="2")

UpdateRelStyle(checkout, stripe, $textColor="orange", $lineColor="orange", $offsetX="-20")
UpdateRelStyle(checkout, inventory, $textColor="blue", $lineColor="blue", $offsetX="-20")
UpdateRelStyle(checkout, email, $textColor="green", $lineColor="green", $offsetX="-20")
