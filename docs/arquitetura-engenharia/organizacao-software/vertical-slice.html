Vertical Slice Architecture

Aqui vamos tratar de uma forma diferente (não quer dizer que seja adequada ao seu projeto) onde todo o código de cada parte do software é agrupado em um único lugar.

Vertical Slice foca em “fatias” completas: cada feature encapsula endpoint, validação, lógica de domínio e persistência, evitando dependências transversais. CQRS separa comandos (mutações) de queries (leituras), otimizando performance e modelagem.

Como exemplo de solução técnica, usaremos Vertical Slice Architecture com CQRS e Mediator em FastAPI, organizando o software por funcionalidades end-to-end, priorizando coesão por feature sobre camadas genéricas.

Conceitos Aplicados

  • Vertical Slice Architecture: Organização do código por funcionalidades (features) em vez de camadas técnicas (controllers, services, repositories). Cada slice contém tudo o que é necessário para realizar uma operação.
  • CQRS (Command Query Responsibility Segregation): Separação clara entre operações de escrita (Commands) e operações de leitura (Queries).
  • Mediator Pattern: Desacopla o remetente da requisição (endpoint) do seu executor (handler), centralizando a lógica de despacho.

Fluxo Conceitual

  1. Endpoint recebe request HTTP e instancia comando/query via schemas Pydantic.
  2. Mediator roteia para handler específico, injetando DB session.
  3. Handler executa domínio/persistência, retorna response serializado. [

Conceito Vertical Slice + CQRS Camadas Tradicionais
Organização Por feature (ex: pasta Product) Por tipo (controllers/services)

Estrutura do Projeto

  • app/core: Contém a infraestrutura compartilhada, como o motor do Mediator customizado.
  • app/features: Diretório raiz das funcionalidades. Cada subpasta (ex: product) representa um slice vertical completo.
  • Componentes do Slice:
    • commands.py/queries.py: Definição dos DTOs de intenção (Data Classes).
    • handlers.py: Lógica de negócio e persistência.
    • endpoint.py: Pontos de entrada da API específicos da feature.
    • schemas.py: Contratos Pydantic para validação de entrada/saída.

Implementação Técnica

  • Mediator Custom: Utiliza um dicionário de mapeamento global para associar tipos de Command/Query aos seus respectivos Handlers.
  • Persistência: SQLAlchemy com padrão SessionLocal e injeção de dependência via FastAPI (Depends(get_db)).
  • Configuração: Uso de pydantic-settings para gestão de variáveis de ambiente e URL do banco de dados.

Vantagens da Abordagem

  • Baixo Acoplamento: Mudanças em uma feature (ex: Produto) não afetam outras slices (ex: Pedido).
  • Alta Coesão: Todo o código relacionado a uma regra de negócio está geograficamente próximo no sistema de arquivos.
  • Escalabilidade de Time: Diferentes desenvolvedores podem trabalhar em slices distintas sem conflitos de merge constantes em arquivos globais.

Desvantagens do Vertical Slice

A metodologia melhora a coesão por feature, mas exige disciplina para evitar divergência estrutural e complexidade desnecessária. Problemas que podem aparecer:

  • Duplicação de código: validações e mapeamentos podem se repetir entre slices.
  • Governança mais difícil: ausência de camadas rígidas aumenta risco de inconsistência entre features.
  • Curva de aprendizado: menos intuitivo para quem está acostumado com arquitetura em camadas.
  • Granularidade sensível: slices mal definidos geram fragmentação ou acoplamento indevido.
  • Regras compartilhadas: difícil decidir entre duplicar ou extrair lógica comum.
  • Testes transversais: fluxos que cruzam múltiplas features exigem mais integração.
  • Overhead inicial: CQRS + Mediator podem ser excessivos em sistemas simples.
  • Debugging indireto: uso de Mediator e behaviors pode ocultar o fluxo real.
  • Baixa enforceability: depende mais de convenções do que de restrições técnicas.

Limite Natural: ainda estamos em um Monolito

Até este ponto, toda a abordagem apresentada, mesmo com Vertical Slice + CQRS + Mediator ainda assume um único processo, um único deploy e, em geral, um único banco de dados. Em outras palavras, trata-se de um monolito bem estruturado.

Essa distinção é importante: organizar o código por features melhora coesão e evolutividade, mas não resolve, por si só, problemas de escala organizacional ou operacional.

Características ainda monolíticas:

  • Deploy único: todas as slices são publicadas juntas
  • Acoplamento em runtime: tudo roda no mesmo processo
  • Banco compartilhado: modelo de dados centralizado
  • Escala vertical predominante: aumento de recursos na mesma unidade

À medida que o sistema cresce, surgem pressões que o modelo monolítico (mesmo bem organizado) não endereça completamente:

  • Escala independente: algumas features exigem muito mais carga que outras
  • Times autônomos: múltiplas equipes precisam evoluir partes do sistema sem coordenação global
  • Deploy desacoplado: necessidade de liberar funcionalidades sem impactar todo o sistema
  • Isolamento de falhas: evitar que problemas em uma feature afetem o todo

Nesse ponto, a limitação não é mais estrutural de código, mas de fronteiras de execução e organização.

O Vertical Slice pode ser visto como um passo intermediário natural para arquiteturas distribuídas:

  • Cada slice já define um boundary funcional claro
  • Commands/Queries já representam contratos explícitos
  • Handlers encapsulam lógica isolada por feature

Isso reduz o custo cognitivo de evoluir para:

  • Microservices
  • Modular Monolith com boundaries fortes
  • Arquiteturas orientadas a eventos

Sistemas de maior escala exigem mais do que organização de código exigem separação física, autonomia de deploy e isolamento de contexto.
